可重入与线程安全

线程安全要求不同线程不能同时访问临界区；而可重入在线程安全的基础上还要求同一线程能够嵌套访问临界区。

狭义而言，线程安全与可重入是相对独立的两个概念。

对于线程安全，通过加锁，我们能通过防止不同的线程同时执行临界区代码来保证；

对于可重入，在单线程情景下，嵌套调用的过程中，局部变量将由调用机制得到保护，但是全局变量则不在此列，这是不可重入的来源。而通过手动保护使用到的非局部变量或干脆避免使用，我们便能够实现可重入。

线程安全的加锁机制，阻止了单线程下实现的可重入，因为虽然我们保证手动保存全局变量，但锁直接阻止了其他线程的访问。

python进程模型

multiprocessing模块提供了一个Process类来代表一个进程，该类提供了一些方法来控制进程的生命周期，比如start、join、terminate等。然而我们该如何理解python的进程与操作系统进程间的关系呢？

首先需要理解python进程的结构，python进程以python解释器为主导，对编译后的字节码进行解释执行。在python中创造子进程，对应地在操作系统内核中也需要创建进程。

进程作为系统资源，是非常敏感的，需要由系统调用来操作，然而不同操作系统提供的系统调用不尽相同，python作为一种跨平台语言，自然需要兼容不同操作系统的进程方案。

在类uinx操作系统中，我们使用fork来创建进程，而在windows操作系统中，我们使用CreateProcess来创建进程，这些差异都被python的Popen库所包装，提供一致的接口。但其中亦非简单地调用系统调用中的fork、CreateProcess，而是又除了最基础的fork外，又追加了spawn与forksever两种方案。

总的来说：

  • fork：fork方案即Unix系统下的最朴素地方案，其写时复制父进程，并从fork()返回，依据返回值判定所处进程，执行不同操作。这种方案执行块(因为是写时复制)，但也存在许多问题，一方面完全复制将共享父进程的文件描述符、套接字等资源，这些资源对于子进程有时是不必要的；另一方面，复制却不会复制父进程的子线程，导致在多线程环境下，子进程中可能存在一些资源导致的死锁问题。
  • spawn：spawn是与fork截然不同的思路，符合Windows下CreateProcess中父子进程耦合低的特点。其主要思路即创建一个全新空白的进程，然后从父进程将需要执行的函数（一般子进程就是为了执行某一个函数）的相关数据通过IPC传入子进程，然后再开始执行。这种方案避免了fork中的高耦合带来的各种问题，但由于执行一个python进程仍旧需要加载python解释器等相关资源，其效率并不高。PS:fork后调用exec即可实现spawn
  • forkserver：forkserver是一种用于在需要多个子进程的情景下，对spawn与fork的结合补充。我们之前提到过，fork速度快，但由于继承父进程资源，问题多；spawn安全，但由于需要加载python环境，较慢。于是，我们可以先spawn出一个干净的forkserver，之后每次需要新的子进程就fork这个forkserver。这样一来，就实现了安全与高效的多进程解决方案。当然，现在一般使用进程池来应对多进程场景，forkserver使用较少。

布隆过滤器

之前看缓存问题时了解到了缓存穿透的部分解决方法，除了一般的对请求进行检验外，还有一种方法即布隆过滤器。布隆过滤器是一种空间效率很高的随机数据结构，它可以用来告诉你，一个元素是否在一个集合中。它的优点是空间效率和查询时间都远远超过一般的算法，缺点是有一定的误识别率和删除困难。

具体来讲，布隆过滤器使用一个bit数组，对于每个键值，使用多个hash函数，并将bit数组中对应索引的几个位置1，这样完成一个键值的插入。

当需要查找时，使用hash函数计算出其对应的几位，在bit数组中查找之，若都为1，则可能存在；若有一位为0，则一定不存在。

布隆过滤器能够实现高效的缓存存在性判断，虽然存在一定的误报率，但只要使用合适长度的bit数组，可以一定程度避免这个问题。当然，其最大问题在于难以支持删除。

dict/map的hash实现与树实现

dict/map是非常常用的数据结构。在python中，dict/map的实现是基于hash的，即使用hash函数将键值映射到一个数组中，然后在数组中查找对应的值。这种实现方式的优点是查询速度快，缺点是空间利用率低，当键值过多时，会出现hash碰撞，导致查询速度变慢，一般使用开发定址或链地址解决。
在java中，除基于hash的hashmap，还存在基于树的treemap，其底层使用红黑树或者平衡二叉树等数据结构来实现。这种实现方式的优点是空间利用率高，缺点是查询速度慢。

mmap内存映射

一句话总结：虚拟内存调度下内存页总与磁盘上的交换文件间来回交换，因而虚拟内存限制了总可用内存的大小。mmap不使用交换文件，而是将虚拟内存页映射到磁盘文件上，这样可以将对文件操作变更为对内存操作，从而提高效率。

WebServer与Spring框架

这一块内容值得分为两块去说：

  1. Web框架
  2. Spring

Web框架

了解MVC之前，我们需要先认识一下WebServer与Web框架。在互联网刚开始的时候，即Web1.0时代，大多网页主要起一个信息展示的功能，基本都是静态网页，通过跳转实现浏览。这时候可以说只有WebServer，而没有Web框架。

到了Web2.0时代，页面元素多媒体化，静态元素装不下；交互需求如评论，得不到满足；前端js兴起，不需要页面跳转。动态Web请求成为主流，Web服务器不再是一个index.html文件的搬运工，而需要动态生成数据。

诚然，我们可以在WebServer中进行实现，但这无疑带来巨大耦合，所以一般单独拆一层出来，即Web框架，用于处理动态请求。这之间呢，就需要一个标准接口对两方进行适配，最初的接口标准叫CGI，即通用网关接口。CGI的工作原理是，WebServer接收到请求后，将请求信息传递给Web框架，Web框架处理后将结果返回给WebServer，WebServer再将结果返回给客户端。这样，WebServer与Web框架之间就解耦了。

在python中，这个接口层叫WSGI、ASGI，而在Java中，对应的就是Servlet这套东西了。

一般使用WebServer，Web框架需要对每一个Servlet单独进行实现，并映射到指定http路径上，之后WebServer就能分派请求，完成整个请求的处理。这种模式下，众多Servlet虽然是Web开发者实现的，然而其却是由WebServer进行管理的，许多需求难以得到满足，比如无法进行同一异常处理等。索性使用dispatcherServlet，从WebServer初取来所有请求，让后自己内部使用众多自定义的Controller实现，这下便可以统一调用数据层、视图层了。

Spring

再说Spring，Spring中除了提供MVC这样的模式之外，还实现了IOC，也就是控制反转，将全局中的大多对象交由一个Bean容器管理，而无需类自己管理。

最早使用xml配置，现在一般使用annotation对需要管理的对象/类进行标识。

  1. @Component：一般用于标识一个类，表示这个类是一个Bean，可以被Spring容器管理
  2. @Bean：一般用于标注在一个返回Bean的方法上，表示这个方法返回的Bean可以被Spring容器管理，常见于对第三方类的管理。
  3. @Service：同Component，表示业务逻辑
  4. @Repository：同Component，表示数据层
  5. @Controller：同Component，表示控制层

WebServer启动后，或是通过实例化Servlet或是通过某个Listener，开始Spring的生命周期。一般会先扫描所有标识了Component的类，然后将其实例化，放入Bean容器中，之后，将自身挂靠在dispatcherServlet下。

谈谈IOC

遥记开始学习编程的时候，我们总与main函数相伴。到后来，接触了Spring、FastAPI等框架，我们就再也没有在main中编写过自己的代码了。

框架为我们实现了程序间共通的一切，网络调用、数据库连接等等，而我们仅需要完成自己程序独立的业务逻辑，等待框架调用即可。程序的控制流不再受自己掌控，而被框架拿捏，这便是Inversion of Control（IOC）。

IOC是一种编程思想，对其的实现多种多样。其中，在Web框架领域最为兴盛的莫过于Dependence Injection与Server Locator。

Dependence Injection即将组件间的依赖交由框架来管理，实现系统中组件间的正确依赖链，而免去手动管理组件生命周期的过程；Server Locator亦是这么一种思想，但仍旧存在组件显示向Server Locator申请依赖对象的过程，故不太被推崇。

此外，模板模式、策略模式，以及UI逻辑等，也是IOC思想的体现。

谈谈变量、内存分配与参数传递

学习一门语言，需要学习其基本语法逻辑。现代通用编程语言，其间的变量定义、循环、分支等逻辑基本都是相通的，但这绝不意味着学会一门语言后能够很快上手其他语言——我们需要在思想中对语言的行为建模，才能深入地理解语言的特性，对语言的种种特性得到一种普适的解释，否则，依靠死记硬背规则、特例进行编程，将给我们带来极大的心智负担，全然不能说是掌握了这门语言。

C

C这门语言的模型可以说是十分面向计算机系统的，我们可以在所有的操作系统课程中了解到一个C语言进程的内存分布，所有语句在编译时、运行时的行为。

Java

Java是对C++的一种简化与改善。

Python

python是一种具有极其简洁设计思想的语言，其中，所有元素，包括基本类型、一般类对象，以及函数，都被建立为对象，存放在堆中，而所有对这些对象的引用被存放在栈中。

由于python中的对象仅能够通过引用进行访问，故其参数传递是完全的传引用。

Go

go语言是我接触的第一门带GC的编译型语言，其对象内存的回收不依赖于一个统筹万物的虚拟机、解释器，而完全通过编译时的判断进行。由于依赖于编译器进行优化，一样的语句在不同程序中完全可能对应着不同的实现。

例如，在Go中声明一个对象，按照C的逻辑，其应该存在于栈上，但假若我持有其指针，离开这个函数，却依旧能访问到这个对象，这对于C是不可想象的，而Go的编译器却能够在编译时判断到这一点，在声明时将其存入堆中。

这样的逻辑模糊了我们的心智模型，但却简化了操作，是好是坏，有待考量。

值得一提的是，Go的传参，与python全然不同，是完全传值的！
