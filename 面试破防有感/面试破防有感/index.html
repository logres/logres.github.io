面试破防有感

发表于 2022-08-24 更新于 2022-08-29 分类于 面试破防有感

昨日参加了字节飞书后端开发——品质保障的面试，一度大为破防，有感，遂记之。

Turn One 自我介绍

虽然料想到会有这么一关，但我还真就是没准备，只能想到啥说啥。姓名、学籍、技能、规划自不能少，但项目我就不主动谈论了——毕竟深度不够，还是不要班门弄斧为妙。

内容虽然都覆盖到了，但顺序明显出了问题，技能竟然被我放在了规划后，不得不说是一项大失误——小寄。

Turn Two 项目介绍

虽然在自我介绍中刻意省去了这一部分内容，但该来的还是要来：“挑一个你觉得最有深度的项目讲讲。”

舍去科研实习与另外两个ppt竞赛的经历，发现最能拿得出手的竟然是一个课程设计。硬着头皮给面试官讲完了我引以为傲的集群设计后，似乎引起了面试官奇怪的兴趣……——小寄。

Turn Three 数据库与缓存

课设中用到了Mysql与Redis，这引发了面试官极大的兴趣，使其以为我是个Mysql与Redis高手——实际上只是会用罢了。于是，他对我发起了夺命连环问：

Mysql单表查询优化

答曰：

  1. 索引
  2. 水平分表（时间片）——还讲了讲字节看到的业务划分例子与自己业务难以划分的问题
  3. 垂直分表（业务）——Mycat-不支持跨表查询

结果：不太满意，想要更细致答案

事后诸葛亮：

待填充

Mysql数据库引擎及其优劣？

答曰:

  1. MyIsam(没念出来)
  2. InnoDB-特性包括事务支持/隔离性支持等
  3. 分布式引擎~

结果：不太满意，我自己也不太满意

事后诸葛亮：

待填充

Redis数据类型

答曰：

  1. String
  2. List
  3. Set
  4. Hash
  5. Zset

再问之：

  1. 知道底层实现吗？不知道
  2. 直接用Redis存储kv还是用Hash？用hash可以更好地管理键值命名空间，避免混乱（不知道答得对不对）

结果：相当一般

事后诸葛亮：

Turn Four TCP网络

  1. 服务端在连接过程中的状态转换？不会
  2. 最大套接字数量?66536，错！
  3. 数据包在公网上的传输过程——一通乱糊，感觉需要个稿子，好好理一理
  4. HTTP版本与理解?1.0短连接 1.1长连接 2.0流式传输+内容压缩 3.0不会
  5. HTTPs中的s？安全套接字层,还行
  6. HTTP状态码？寥寥数个
  7. 浏览器框边上的锁?不会

直接寄！

事后诸葛亮：

  1. TCP状态机
  2. 考虑客户端——端口限制/服务端——min(四元组限制2^48^,文件描述符数量限制——可设置)
  3. ……
  4. 2.0的特性与谷歌的QUIC HTTP3.0
  5. 感觉可以更详细地记录一下过程
  6. 状态码
  7. 查看站点信息，包括cookie、证书

大寄！

Turn Five 系统

  1. 进程描述——勉强吧
  2. 进程/线程/协程——这个我会
  3. 死锁——这个我会
  4. 守护进程——开在后台，不受前台控制影响-为init的子进程，可以在断开终端后继续运行
  5. 僵尸进程——忘了！（答案：未被父进程及时回收的结束子进程)
  6. 最大进程数量：一般受内存大小影响-无其他限制

小寄！

Turn Six 做题

最大不重复子字符串！

我的解法：哈希加暴力搜索$O(n^2)$
最优解发：滑动窗口$O(n)$

中寄！

Turn Seven 反问

这一阶段我终于搞懂了这个岗位的业务需求了！

  1. 工作内容：
    问曰：品质保障干些什么？
    答曰：保障跨国跨地公司远程应用体验。
    再问曰：那就是假设内容分发网络之类的咯？
    答曰：假设cdn、服务器等等
  2. 服务模式：
    问曰：主要是为公司提供Rest API接口，由用户公司自行搭建平台吗？
    答曰：API主要用于帮助用户融合自身平台
  3. 数据的管理：
    问曰：使用字节的网络API，其余公司对自身数据安全的想法？
    答曰：主要提供技术支持，用户自行搭建服务器提供服务

总结

寄中寄！只希望面试官给个机会，不然就让感谢信来的快一些吧！
